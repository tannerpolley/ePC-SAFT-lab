# Source-Faithful Equilibrium Recovery And Curated Migration Program Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> `superpowers:subagent-driven-development` or `superpowers:executing-plans`
> only after selecting one stage and writing or approving its bounded child
> plan. Use `chemical-engineer` for every thermodynamic or numerical stage,
> `superpowers:test-driven-development` for implementation slices,
> `superpowers:systematic-debugging` for unexpected failures, and
> `superpowers:verification-before-completion` before every checkpoint.

**Goal:** Recover a source-faithful, compact equilibrium architecture from the
valuable current implementation, then transfer only accepted owners and
evidence into a curated repository set.

**Architecture:** Stabilize doctrine and ownership before changing code.
Preserve one provider-resolved model boundary, one phase evaluator, one local
NLP interface, one Ipopt adapter, typed problem formulations, a distinct HELD
controller, and native-owned certification. Execute the program as thirteen
independent stage checkpoints; this program plan coordinates them and never
authorizes all thirteen as one implementation batch.

**Tech Stack:** C++17, pybind11, CppAD, Ipopt, a Stage II LP backend selected by
the neutral HELD child design, Python 3.13 development baseline, NumPy, pytest,
`uv`, Ruff, Sphinx, CMake source manifests, retained Markdown literature, and
Git/GitHub milestone mirrors.

## Global Constraints

- Preserve public routes exactly until a separate admission plan passes:
  `bubble_pressure`, `dew_pressure`, and scoped methane/ethane/propane
  `single_component_vle`.
- Neutral LLE, neutral TP flash, generalized multiphase, electrolyte LLE,
  standalone CE, reactive LLE, reactive electrolyte LLE, and CPE remain closed
  unless their independent stage and admission gates pass.
- Use direct local commits on `main` only when the active stage explicitly
  authorizes them and the exact diff is reviewed. Never push without explicit
  user authorization.
- Preserve stashes, branch-only scientific work, retained plots, and failed
  evidence until their content is independently classified.
- No fallbacks, compatibility wrappers, duplicate owners, fake defaults,
  invented parameters, or paper-specific branches in installed numerical
  code.
- Finite starts and sampled phase candidates do not establish global phase-set
  completeness.
- Tests computing model predictions use traceable real literature inputs and
  retain literature-versus-model plots. Algorithm unit tests may use analytic
  toy free-energy surfaces when the test explicitly targets mathematics rather
  than a scientific capability claim.
- New or updated plots must be rendered in the stage handoff with absolute
  paths and a compact retained-data table.
- Each implementation slice begins RED, changes one owner or behavior, proves
  GREEN, removes the displaced path, and commits a focused checkpoint.
- Run `bash "$HOME/.codex/hooks/codex-cleanup.sh" --repo-root .` before every
  stage handoff. Remove only task-owned ignored artifacts.
- Do not run full paper campaigns, all wheel combinations, or full collection
  for a slice that does not own those outputs. Use the verification ladder in
  this plan.

## Outcome Proof

**Intent:** Recover and preserve the scientifically valuable equilibrium work
without carrying forward misleading algorithm claims, duplicated owners, or
paper-specific runtime structure.

**Current Behavior:** The provider is comparatively mature, while equilibrium
contains exact local mechanics alongside incomplete HELD/HELD2 orchestration,
overlapping route owners, broad diagnostic bindings, and mixed M4/M6 evidence.

**Expected Outcome:** A compact provider-to-equilibrium boundary, canonical
local solver kernel, consolidated public routes, source-faithful neutral and
electrolyte algorithm plans, separated CE/CPE work, and a curated
multi-repository migration path under the `ePC-SAFT` GitHub organization with
explicit preservation provenance.

**Target Output:** Thirteen accepted stage receipts, corrected doctrine and
tracker state, preservation manifest, stable resolved-input SDK, minimal
equilibrium kernel, canonical public-green routes, source-faithful algorithm
controllers when implemented, per-repository curated migration manifests,
cross-repository installed-artifact receipt, and final retirement receipt.

**Owner:** M0 sequences the program and owns repository-home governance; M3
owns resolved model input and EOS validity; M4 owns algorithms, local NLPs,
activation, Ipopt, and certification; M5 owns regression and its admission
blockers; M6 owns literature data and reproduction evidence; M1 owns package
artifacts and Stage 13 package-boundary proof.

**Interface:** `ResolvedNativeInput` or its accepted M3 successor,
provider-backed `PhaseEvaluator`, `LocalNlpProblem`, the shared Ipopt adapter,
typed route requests/results, native activation descriptors, HELD controller
receipts, and M6 evidence manifests.

**Cutover:** Each stage moves callers directly to the accepted owner and
deletes the displaced path in the same slice. Curated-repository transfer is
file/owner selective and records original commit plus acceptance receipt.

**Replaced Path:** Loose provider payload construction, equilibrium-side model
defaults, duplicate direct/activated route dispatch, sampled-candidate dual
claims, residual-only Stage III identity, paper fingerprints in runtime,
counterion-pair code labeled as Perdomo HELD2, and closed diagnostic production
bindings.

**Evidence:** Source-anchor tests, ownership characterization, exact derivative
tests, fresh native builds, focused API/native suites, algorithm unit oracles,
selected source-backed reproduction, package confidence, docs, Ruff, diff,
cleanup, and independent scientific/code review.

**Acceptance Proof:** A stage passes only when its named owner and behavior are
proven, displaced paths are absent, public capability state is unchanged unless
explicitly admitted, and its checkpoint receipt is reproducible from current
source.

**Stop Criteria:** Stop on unresolved governing equations, stale native
identity, missing scientific inputs, unapproved capability change, a second
live owner, cross-stage scope expansion, unexplained numerical drift, failed
focused proof, or a need to repair individual paper programs outside the
stage's M6 scope.

**Avoid:** Do not execute the thirteen stages as one branch, mechanically split
large files, preserve wrappers for convenience, tune solvers before verifying
inputs, or use full-suite volume as a substitute for focused scientific proof.

**Risk:** Source cleanup can erase valuable internal mathematics, while broad
compatibility can recreate the current density. Characterization-first
cutovers, small stage commits, exact source identity, and independent review
control both risks.

## Implementation Boundaries

**Files To Create:** Stage-owned doctrine, characterization, controller,
problem, certificate, and migration-manifest files named by each child plan.
This program plan does not pre-authorize implementation filenames where the
stage design has not selected the interface.

**Files To Modify:** M4/M6 doctrine and trackers in Stages 1-2; provider input
owners in Stage 4; equilibrium kernel and public routes in Stages 5-6; selected
algorithm owners in Stages 7-10; repository-home governance in Stage 11;
package/repository metadata and proven owners in Stage 13; retirement/archive
records in Stage 12.

M6 stage, regression implementation, release credentials, downstream private
projects, retained stashes, and unrelated generated artifacts.

**Source Of Truth:** Local literature markdown for algorithm identity,
`docs/latex/equations.tex` for EOS equations, M3 resolved-input contract for
model state, native activation matrix for exposure, and stage-owned tests for
accepted behavior.

**Read Path:** Source doctrine -> stage child design -> RED characterization or
behavior test -> implementation owner -> focused proof -> stage receipt.

**Write Path:** Select one stage, refresh baseline, approve its child design,
write RED proof, implement the smallest owner change, delete displaced paths,
verify, review, commit, and stop before the next stage.

**Integration Points:** Provider SDK, phase evaluator, CppAD derivative blocks,
Ipopt adapter, selected LP backend, native activation, result builder, Python
request/result surface, M6 evidence registry, CMake manifests, and package
build profiles.

**Migration Or Cutover:** Stage 11 creates governance-only local repository
homes after Stage 3. Stages 4-6 then prepare the stable core, and Stage 13
performs its bounded proven-owner transfer and separately approved
development-source cutover. Stages 7-10 are independent optional branches from
Stage 6 and require separate transfer leaves when accepted. Stage 12 alone owns
transition-repository retirement/archive; the optional validation repository
remains deferred pending a separate ownership decision.

**Replaced Path Handling:** Delete superseded runtime paths after the last
caller moves. Preserve historical documents with explicit supersession notes
or archive placement; do not silently rewrite history.

**Acceptance Proof Gate:** Slice proof -> boundary proof -> stage proof.
Admission, release, or final closeout additionally requires the exact broader
gates owned by that claim.

## Program Handoff Rule

A future agent must select one task above, refresh its baseline, and finish or
stop that task before advancing. “Continue the program” never means executing
multiple scientific stages without their individual review gates. At each
handoff report:

- active stage and exact checkpoint commit;
- changed owners and deleted paths;
- focused and stage-level validation results;
- capability changes, normally none;
- deferred failures by exact test/checker name and cause;
- worktree, branches, stashes, and remote state;
- next stage entry conditions.
